## 关于这个 App

原版 LOOI 应用会把一些功能锁在服务端配置和硬件传感器之后。通过有针对性的 Smali 代码修改，我已经成功绕过了其中一部分限制。作为对我这项工作的回应，从 2.6.1 版本开始，TangibleFuture 使用了付费版（而且不便宜）的 Jiagu 360 加固和混淆方案，使这些版本的代码更难访问。Jiagu 360 在更早的版本中其实也已经存在，但由于其配置方式，当时并没有对代码访问造成实际影响。

就在今年，TangibleFuture 团队重新评估后，决定给我开放 App 开发版本的访问权限，这让我相比受保护版本的工作流，能够更快地进行调试和开发。

### 绕过更新检查

为了永久禁用强制更新机制，并防止应用卡死在启动页，我修改了 `UpgradeManager$tryUpgrade$2.smali` 中的版本比较逻辑。原始代码使用条件分支指令 `if-le`，将服务器版本号与本地版本进行比较。我把这条指令替换成了无条件跳转 `goto`，强制执行流程始终绕过“发现可用更新”的逻辑。这样应用每次都会直接走“无需更新”的回调路径，无论服务器上的实际版本是多少，启动页都能正常关闭。  
这意味着你可以运行任意版本的 App，而不用担心更新问题。

### 解锁未完成的游戏（LOOI Pong）

应用资源中包含一个完整可运行的 `LooiPingPongActivity`（乒乓球）页面，但它在主菜单的 `GameCenterActivity` 中被隐藏了。需要说明的是，这个游戏本身是损坏且未完成的，本质上只是一个双人 Pong。它大概是因为双人脸识别系统性能过低而被废弃的。

**修改内容：**

* **目标：** `com.TangibleFuture.looiRobot.game.activity.GameCenterActivity`
* **菜单注入：** 我把 `initData` 中 `GameBean` 数组的大小从 3 扩展到 4，并注入了一个新的游戏项。
* **逻辑：** 启动函数 `jumpGame` 会检查一个特定的本地化字符串。我把菜单项名称硬编码成 **"LOOI Pong"**（因为这里还有一些我暂时没完全搞明白的奇怪函数限制），并修改了 `jumpGame`，让它接受这个字符串作为启动隐藏 Activity 的“密码”。

### RC Unleashed

原版应用会通过监控 `phone_stick_state` 变量（表示磁吸安装状态）来人为限制遥控功能。如果检测到手机“安装”在机器人上，App 就会弹出强制性的 “Take Down Phone” 对话框并阻止进入遥控界面，逼你把它的脑袋拆下来。可问题是，如果你就是想在上面装个纸板 LOOI 再开呢？

**修改内容：**

* **目标：** `com.TangibleFuture.looiRobot.activity.RemoteControlActivity`
* **绕过逻辑：** 我定位到了 `onResume` 生命周期方法中的传感器检查逻辑，并移除了会触发拦截弹窗的条件跳转。同时，我还把 `showTakeDownPhoneDialog` 方法“废掉”了，避免异步触发。
* **结果：** 无论手机当前是否安装在机器人上，RC 遥控界面都会立即加载，从而实现安装状态下依然可以正常操控。

## 📡 关于 LOOI 的 BLE 协议

LOOI 使用的是一种基于序列的 BLE 协议。它并不是简单的直接遥控驱动，而是采用了一种“脚本模式（Script Mode）”逻辑，其中命令会作为动画序列中的帧来执行。

### 连接握手

遗憾的是，你不能只是用 NRF Connect 连上设备然后立刻开始发命令。机器人要求由官方 App 发起一段握手序列（据我所知，本质上只是让机器人输出它可用的组件信息）。如果缺少这段握手，LOOI 会在几秒后断开连接。伪装一个假的电机底座也是可行的。

先通过官方 App 连接机器人，把 App 切到后台（不要杀掉），然后再用 NRF Connect 连接。 :-)

### 命令接口（`fe00`）

主写入特征值是 `fe00`。它接受多字节字符串，用来协调电机、LED 和屏幕动画。  
命令结构**通常**是一个 **17 字节数据包**。

**参考命令（旋转并将 LED 变红，这是我自定义的命令，欢迎自己试）：**  
`00 07 00 FF 05 00 00 00 00 64 02 0A 96 02 14 00 02`

**数据包拆解：**

| 字节索引 | 值（Hex） | 字段名 | 说明 |
| --- | --- | --- | --- |
| **0** | `00` | **SEQ** | 循环序列计数器（00-FF）。必须递增，否则数据包会被丢弃。 |
| **1** | `07` | **OPCODE** | 命令类型，具体含义我还在继续确认。 |
| **2** | `00` | **SUB_OP** | 子命令或模式修饰符。 |
| **3** | `FF` | **MASK_A** | 目标系统掩码 A。 |
| **4** | `05` | **MASK_B** | 目标系统掩码 B。 |
| **5-8** | `00..00` | **PAYLOAD** | 4 字节数据块。 |
| **9** | `64` | **VALUE** | 幅值/速度。`64`（Hex）= 100（十进制）。 |
| **10** | `02` | **PARAM_1** | 辅助参数 1。 |
| **11** | `0A` | **PARAM_2** | 辅助参数 2。 |
| **12** | `96` | **DURATION** | 执行时长，单位为 tick/ms（`96` = 150）。 |
| **13** | `02` | **PARAM_3** | 与上下文相关。 |
| **14** | `14` | **PARAM_4** | 与上下文相关。 |
| **15** | `00` | **RES** | 保留字段。 |
| **16** | `02` | **END/CRC** | 尾部或校验值。 |

### 已知特征值与 UUID（未完成）

| UUID 前缀 | 功能 | 说明 |
| --- | --- | --- |
| **fe00** | **命令接口** | 动画和运动的主写入通道。 |
| **ff02** | **电机驱动（Boost）** | 高速运动。发送 `FF` 据说会改变方向。 |
| **fed0** | **电机驱动（Slow）** | 精细运动控制。 |
| **fed1** | **颈部电机** | 头部独立运动控制。 |
| **fed2** | **头灯/手电** | 直接控制：`00` = 关，`03` = 开。 |
| **fed9** | **传感器数据** | 只读数据流（悬崖传感器、TOF 距离、电池）。 |
