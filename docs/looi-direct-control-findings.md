# LOOI 直控通道结论

## 结论摘要

- 官方 App 的“自由遥控”主通道不是 `fe00`。
- 当前已确认：
  - 底盘连续控制走 `fed0`
  - 头部连续控制走 `fed1`
  - 大灯开关走 `fed2`
- `fe00` 仍然存在，但更像：
  - 一次性动作脚本
  - 姿态/模式切换
  - 某些进入 RC 态时的辅助包

这解释了一个此前的现象：

- 用 `fe00` 的既有样本可以触发“低头”“旋转”“前冲”这类单次动作
- 但不能稳定实现“像摇杆一样一直前进 / 一直左转 / 一直右转”

## 样本来源

- HCI：
  - `work/hci_20260624_015528/common/com.android.bluetooth/btsnoop_hci_260624_015443.log`
- 用户本轮官方 App 操作顺序：
  1. 上
  2. 左
  3. 下
  4. 右
  5. 从抬头到低头的连续头部指令
  6. 开大灯
  7. 关大灯

## 握手后进入 RC 态的辅助包

在这轮样本里，握手后先看到：

- `01:54:49.618`
  - `fe00 = 00010000010032030a0001ff00010a3203ff0003`
- `01:54:49.645`
  - `fed2 = 00`
- `01:54:50.139`
  - `fe00 = 00020000010032030a0001ff00010a3203ff0003`

这两帧 `fe00` 更像“进入某种 RC/姿态状态”的辅助写入，不像直控本体。

## 底盘连续控制：`fed0`

### 通道与写法

- characteristic: `fed0`
- handle: `0x004e`
- ATT opcode: `0x52`
- 类型：`Write Command`
- 单包长度：`2` 字节

这很关键：

- 官方 RC 在底盘移动时，不是发 17/20 字节 `fe00`
- 而是在很短间隔内，持续发 2 字节 `fed0`

### 本轮 4 组底盘控制簇

#### 1. 第 1 组：`01:54:55.635 -> 01:54:56.927`

推定对应用户第 1 个动作：`上`

代表值：

- `542a`

- `720c`
- `7905`
- 稳态附近：
  - `7707`
  - `7608`
  - `7509`
- 结束：
  - `0000`

粗看像是先快速推向目标，再在目标附近抖动采样，最后回中。

### 当前可先拿来做实验的 `fed0` 方向稳态值

这些值不是“最终协议定义”，但足够作为下一轮 Web / 原型实验的第一批候选：

- 前：`7707`
- 左：`047a`
- 后：`86fc`
- 右：`0082`
- 停：`0000`

注意：

- 这些值是从一段连续轨迹里摘出的“近稳态值”
- 还不能保证“每隔 N ms 重复发这个值”就一定等价于官方摇杆
- 但这比继续拿 `fe00` 动作包硬猜要靠谱得多
- 用户实机验证后，已确认需要把原先的“后 / 右”候选互换：
  - `86fc` 才是后退
  - `0082` 才是右转

## 头部连续控制：`fed1`

### 通道与写法

- characteristic: `fed1`
- handle: `0x0051`
- ATT opcode: `0x52`
- 类型：`Write Command`
- 单包长度：`1` 字节

### 本轮头部轨迹

用户执行的是“从抬头到低头的持续指令”，抓到两段非常明显的 `fed1` 流：

#### 抬头方向：`01:55:10.194 -> 01:55:10.372`

- `5e`
- `59`
- `49`
- `3d`
- `21`
- `11`
- `02`
- `00`

这段看起来像逐步向一个极限位置推进，最后接近 `00`。

#### 低头方向：`01:55:12.233 -> 01:55:12.499`

- `02`
- `11`
- `23`
- `3f`
- `54`
- `63`
- `84`
- `9e`
- `aa`
- `bc`
- `d1`
- `e8`
- `ef`
- `ff`

之后 `01:55:15.082 -> 01:55:15.245` 又出现一段回拉：

- `f9`
- `db`
- `c9`
- `b2`
- `a8`
- `99`
- `91`
- `89`
- `87`

当前最保守的结论是：

- `fed1` 的 1 字节值就是头部俯仰位置/目标值
- 小值更接近一端，大值更接近另一端
- `00` 与 `ff` 很可能分别对应两个俯仰极限

## 大灯控制：`fed2`

### 通道与写法

- characteristic: `fed2`
- handle: `0x0054`
- ATT opcode: `0x12`
- 类型：`Write Request`

### 本轮确认

- `01:54:49.645`
  - `fed2 = 00`
- `01:55:16.783`
  - `fed2 = 01`
- `01:55:18.586`
  - `fed2 = 00`

结论：

- `fed2 = 01`：开大灯
- `fed2 = 00`：关大灯

这和之前的“`00` 关灯”一致，但把“开灯值”从猜测推进到了 HCI 坐实。

## 当前协议模型

目前最靠谱的控制层分工如下：

- `fed0`
  - 底盘直控
  - 高频 2 字节流
- `fed1`
  - 头部直控
  - 高频 1 字节流
- `fed2`
  - 大灯开关
  - 简单开关量
- `fe00`
  - 高层动作脚本 / 一次性姿态动作 / 进入某模式的辅助包

## 2026-06-24 最新补充：吸附事件不走直控流

基于最新 `adb` 拉下来的：

- `work/adb_pull/bugreport-2026-06-24-124313.zip`

又确认了一件重要的事：

- “手机放上传感器后”的回调链虽然真实存在
- 但它下发给机器人的主体不是 `fed0/fed1/fed2`
- 而是一组新的 `fe00` 动作脚本

系统与 App 侧时间线：

- `12:42:42.501`
  - 系统：`dockSwitch = true`
- `12:43:03.671`
  - App：`吸附状态调试，当前已吸附，设备状态：true`
- `12:43:04.835`
  - App：`robotAction=028202`
- `12:43:05.368 -> 12:43:05.521`
  - HCI：连续 5 帧 `fe00`

对应 BLE 包为：

1. `0000000002003203020002023203051e02092802`
2. `01000b5003101e02130002145f0332000235eb02`
3. `020039eb023c00024500024814024f1402520002`
4. `03006c00026e5f0370e10274460375d7027ae102`
5. `04007d0002ff00027d3203ff0003`

后续收尾：

- `000001ff0001`
- `000001ff0001`
- `00030000010032030a0001ff00010a3203ff0003`
- `fed2 = 00`

这条样本的意义是：

- `fed0/fed1/fed2` 仍然是“自由遥控 / 连续控制”的主通道
- 但“吸附事件”“模式切换”“状态反应”这类高层行为仍然可能继续走 `fe00`
