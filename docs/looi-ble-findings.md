# LOOI BLE 已确认结论

## 目标

本文档沉淀当前已经较稳定、可复用的 LOOI 官方 Android App BLE 连接结论。

- 不记录每一步排查流水。
- 只记录后续实现自定义连接器时会反复引用的事实、映射、时序和缺口。

## 设备与样本

- Android App：
  - 包名：`com.TangibleFuture.looiRobot`
  - 版本：`2.8.1`
  - `versionCode = 48`
- 机器人 BLE：
  - 设备名：`LOOI Robot`
  - 地址：`EC:64:C9:AA:54:96`
- 手机：
  - 型号：`24129PN74C`

## 高层判断

- 单独使用 NRF Connect 连接机器人时，连接通常会在约 3 到 5 秒内被断开。
- 官方 App 连接时，能稳定维持更长时间。
- 官方 App 先完成一段初始化流程后，NRF Connect 可以和官方 App 并存连接数分钟。
- 因此，问题核心不是“能不能建链”，而是“官方 App 在建链后做了哪些最小必要握手”。

## 已确认的连接外层时序

- 官方 App 的连接外层顺序已经多次被系统日志确认：
  1. 扫描 `LOOI Robot`
  2. 建立 LE 连接
  3. 完成链路加密
  4. `discoverServices()`
  5. 拉取远端 GATT DB
  6. 读取若干初始化 characteristic
  7. 订阅两个通知通道
  8. 连续进行 3 次初始化写入
  9. 收到初始化通知
  10. 进入后续运行期数据收发

## 已确认的最小握手骨架

当前最可信的“最小可用握手骨架”如下：

1. 扫描 `LOOI Robot`
2. 连接并等待 LE 加密成功
3. 拉取 GATT DB
4. 读取若干初始化 characteristic
5. 订阅：
   - `fed9` 的 CCCD
   - `fef0` 的 CCCD
6. 写入：
   - `feda` 1 字节
   - `fef0` 20 字节
   - `feda` 2 字节
7. 等待通知：
   - `fed9`，常见长度 3
   - `fef0`，至少见过长度 1
8. 后续运行期还会继续向 `fe00` 写 20 字节包

基于 `2026-06-24 00:55` 这轮 HCI 抓包，本轮实际看到的写入顺序是：

1. 写 `feda` 1 字节
2. 写 `fef0` 20 字节
3. 写 `feda` 2 字节

这和更早一轮仅靠系统日志观察出来的 `feda(1) -> feda(2) -> fef0(20)` 不完全一致。

- 当前更稳妥的结论是：
  - 已确定这 3 次写都存在
  - 至少观察到过两种时序表现
  - 在自写连接器里，优先先按 HCI 直出的顺序复现

## 角色判断

- `feda`
  - 更像连接后立刻执行的短初始化控制口。
- `fef0`
  - 更像连接完成后的大包数据口之一，至少握手期会收到 20 字节初始化写入。
- `fe00`
  - 不是唯一主写口。
  - 但在连接完成后的运行期确实会继续承载 20 字节写入。
- `fed9`
  - 是握手期最稳定出现的通知通道之一。

## 2026-06-24 新增直控结论

基于：

- `work/hci_20260624_015528/common/com.android.bluetooth/btsnoop_hci_260624_015443.log`

现已确认官方 App 的“遥控/连续控制”并不主要走 `fe00`，而是分成 3 条更直接的通道：

- `fed0`
  - 底盘连续控制
  - `Write Command`
  - 单包 2 字节
- `fed1`
  - 头部连续控制
  - `Write Command`
  - 单包 1 字节
- `fed2`
  - 大灯开关
  - `Write Request`
  - 已确认：
    - `00 = 关`
    - `01 = 开`

这说明：

- `fe00` 更像高层动作脚本 / 一次性动作 / 某些模式切换辅助包
- 不适合作为“自由驾驶”的第一候选通道

更细的样本和时间对照见：

  - `0x003d` -> `3` 字节

  - `0x005d` -> `4` 字节
  - `0x0060` -> `2` 字节

  - `0x0072` -> `1` 字节

## 已确认的多次会话特征

- `06-24 00:29:10` 左右，官方 App 会话为 `app_if = 94`
- `06-24 00:30:41` 左右，官方 App 重连后变为 `app_if = 95`
- `06-24 00:41:33` 左右，用户再次复现后变为 `app_if = 96`

这说明：

- “杀后台再重开官方 App”会稳定产生新的 GATT client 实例。
- 连接不是复用旧 session，而是重新走扫描、连接、加密和后续初始化流程。

## 当前最大缺口

当前最大的缺口已经从“payload 未知”缩小为“payload 语义未完全解释”：

- `fef0` 的 20 字节初始化写入里：
  - 为什么是 ASCII 字符串
  - 前导 `A` 的语义是什么
  - 是否每次都需要带当前时间
- `fe00` 的 20 字节运行包：
  - 是否需要周期发送
  - 每个字节位的真实含义是什么
- `fed9` 的 3 字节通知：
  - 是否是传感器状态流
  - 每个字段如何映射到业务状态

## 对实现的直接启发

如果后续要自写连接器，最合理的起点不是直接从 `fe00` 猜包，而是：

1. 精确复现官方 App 的订阅顺序：
   - `fed9`
   - `fef0`
2. 再回放本轮 HCI 已确认的 3 个初始化写入：
   - `feda = 03`
   - `fef0 = 4132303236203036203234203030203535203432`
   - `feda = 8101`
3. 然后观察：
   - 是否能稳定收到 `fed9` / `fef0` 通知
   - 是否还会在 3 到 5 秒内被踢下线
4. 如果能稳定维持连接，再验证：
   - `fe00 = 00000000010032030a0001ff00010a3203ff0003` 是否需要继续发送

### 为什么可以不用 HTTPS

Web Bluetooth 需要的是 secure context，不一定必须是公网 HTTPS。

- `http://127.0.0.1`
- 也就是浏览器认可的 localhost 安全上下文

因此，第一轮本地验证优先推荐：

- `python3 -m http.server`
- `adb reverse`
- 手机 Chrome 打开 `http://127.0.0.1:8000`

- 断开后重连这条路径比首次连接更脆弱
- 已经在验证器里补了：
  - 断开时清理本地状态
  - 阻止并发点击
